# 自适应工作流程设计

## 目标

建立一套适用于日常任务的通用工作流程，并为当前个人网站补充专属检查。流程以效率和可靠性平衡为目标：低风险任务直接推进，中风险任务先给简短方案，高风险任务在执行前获得确认。

## 核心原则

- 先理解真实目标和现有执行路径，再决定修改方式。
- 优先复用现有实现、标准能力和已安装工具，避免不必要的依赖与抽象。
- 只修改完成当前任务所需的范围，不顺带进行无关重构。
- 保留用户已有改动，不擅自撤销、覆盖或清理。
- 完成必须有与风险相称的最新验证证据，不能只以“代码已写完”为标准。
- 可以安全推断时继续执行并说明假设；推断会明显改变结果时才询问。

## 风险分级

风险由影响范围、可逆性、数据与安全影响、是否涉及生产部署共同决定。任一维度达到高风险时，任务按高风险处理。

### 低风险

适用于解释、查询、文案调整、小范围样式修改和目标明确的单点修改。

流程：理解现状 -> 直接执行 -> 快速验证 -> 汇报。

### 中风险

适用于新增功能、多文件修改、交互调整、构建配置变化和可能影响相邻功能的修复。

流程：理解现状 -> 给出简短方案 -> 执行 -> 完整验证 -> 汇报。

### 高风险

适用于删除或覆盖数据、安全认证、依赖升级、生产部署、大范围重构和难以回退的操作。

流程：理解现状 -> 明确方案与影响 -> 获得确认 -> 分步执行 -> 回归验证 -> 交付。

## 通用执行流程

1. 明确目标：确认任务、限制、预期结果和完成标准。
2. 检查现状：读取相关文件、项目说明、近期变更和工作区状态。
3. 选择能力：只启用对任务直接有帮助的技能、工具和已有项目能力。
4. 判定风险：决定直接执行、提供简短方案或等待明确确认。
5. 定位根因：理解调用关系和数据流，避免只修复表面症状。
6. 最小实施：复用现有模式，以最小正确改动完成目标。
7. 分层验证：按风险运行针对性检查、自动测试、构建和必要的人工验收。
8. 复查差异：检查意外文件、无关修改、敏感信息和潜在回归。
9. 清晰交付：说明结果、关键文件、验证证据和仍存在的限制。

## 按任务意图调整动作

- 回答、解释、评审和状态报告以读取与分析为主，不默认修改文件或外部状态。
- 诊断任务负责定位并说明根因；只有用户同时要求修复时才实施改动。
- 修改或构建任务在风险分级后实施、验证并交付完整结果。
- 监控或等待任务持续观察目标状态，状态暂时未变化不视为失败。
- 用户原始请求中已经明确提出的操作视为已授权；实质性扩大范围或新增外部影响仍需另行确认。

## 当前网站专属流程

### 内容和数据

- 检查 `content.json`、归档、作品和媒体数据之间的一致性。
- 数据修改前保留原始内容，避免不可恢复的覆盖。
- 验证空数据、缺失字段和无效资源不会导致页面失效。

### 本地编辑服务

- 涉及服务端逻辑时，检查编辑、发布、删除、评论和上传的相关路径。
- 涉及认证时，不暴露 `.editor-auth.json`、密码或环境变量内容。
- 对外部输入保留必要的验证、路径限制和错误响应。

### 页面和交互

- 检查桌面端与移动端布局、文字溢出、交互状态和资源加载。
- 视觉改动需要浏览器截图或人工检查，不能只依赖单元测试。
- 确认公开静态页面在编辑服务不可用时仍能正常浏览。

### 构建和部署

- 运行现有静态构建流程，确认 `dist` 产物生成且关键页面与资源存在。
- 部署配置变化时检查 `vercel.json`、环境变量要求和敏感文件排除规则。
- 生产部署属于高风险操作，执行前明确目标环境和回退方式。

## 验证策略

- 低风险：运行最接近修改点的快速检查，并复查差异。
- 中风险：运行相关测试和构建；涉及界面时增加桌面端与移动端检查。
- 高风险：在中风险检查基础上增加回归测试、数据保护检查和回退验证。
- 验证命令必须在交付前重新运行；不使用过去的结果作为完成证据。
- 无法执行某项验证时，交付中明确说明未验证内容、原因和残余风险。

## 异常处理

- 测试失败时先定位根因，不通过删除测试、放宽断言或隐藏错误来交付。
- 权限、网络或环境受限时，先尝试安全且等价的替代路径。
- 发现无关问题时记录但不扩大任务；只有直接阻塞当前目标时才处理。
- 用户已有修改与任务冲突时，优先兼容；无法安全兼容时停止并询问。
- 任何未在原始请求中明确提出的破坏性、外部可见或难以回退操作都需要另行授权。

## 交付格式

每次交付优先回答四件事：

1. 做了什么。
2. 修改了哪些关键文件。
3. 如何验证以及验证结果。
4. 尚未完成、无法验证或需要注意的事项。

简单任务用一段简短说明即可；复杂任务再使用分段和列表，避免把过程日志当成交付内容。

## 可选改进

以下改进有价值，但当前不作为强制步骤：

- 为静态构建增加轻量的产物完整性检查，自动确认关键 HTML、JSON 和资源文件存在。
- 为主要公开页面建立少量桌面端与移动端视觉基线，用于识别明显布局回归。
- 为内容 JSON 增加结构校验，尽早发现缺失字段和错误类型。
- 在部署前生成简短检查清单，记录构建结果、目标环境和回退方式。

这些能力应在重复问题出现或维护成本明显增加时逐项引入，避免一次性建设过重的流程系统。

## 成功标准

- 小任务不会因流程本身产生明显等待。
- 中高风险修改在执行前具有足够的边界和影响说明。
- 每次完成都有新鲜、可复核的验证证据。
- 用户现有改动、内容数据和敏感信息得到保护。
- 网站在本地编辑、静态浏览和部署三种场景下都有对应检查。
